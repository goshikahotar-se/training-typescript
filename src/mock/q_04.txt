2. List 2 test cases you would keep manual (or not automate in E2E) and why.
Basically we keep those tests manual - where the test are one-off (would not required to be test after doing once), OR tests where there is too much unstability - for e.g. in this feature I can see:
 i) the the registered users, we do not need to create one from scratch for this feature. We can use existing registered user and do the transfer.
 ii) we also do need need to create new accounts for the registered user each time; we can use existing ones to complete the feature coverage of the transfer.

 //Exploratory — odd inputs, copy/paste account numbers, keyboard/mobile layout

3. Name 3 risks if you only run happy-path E2E in Chromium and skip everything else.
    i) Different user-experience across the different browsers (layout inconsistencies, broken UI)
    ii) Compatibility issues - certain feature might now work
    iii) Performance difference across different browsers.

4. What do you need from BA, developers, and DevOps before you start writing automation?
From BA, we need the actual feature requirements, Acceptance Criteria, edge cases - this will permit us to create our test data accordingly to perform our feature tests.
From developers, we need the technical documentation, API Spec(Swagger) - how the system has been designed.
From Devops, we need our test suites to run on CI - so intergrating the workflow of our E2E our Acceptance suites in the CI is important. Storing our secrets and access keys securely on Github secrets - these help are needed.


“I’d automate transfers mainly at API level for amounts, errors, and balances, plus a thin E2E smoke on Chromium and expand browsers in CI. 
Manual stays for exploratory and MFA-like flows. 
I need clear acceptance criteria, test accounts with known balances, selectors, and a stable QA pipeline with secrets and reports.”
